# Multi-GPU / Distributed Training — Design Notes

> **Status:** not implemented. This document describes the intended design for a
> future Accelerate/DDP extension. The current codebase runs on a single GPU and
> that remains the default and only supported path.

---

## Why naïve parallelism breaks GRPO

Running two independent train processes is **not valid** for GRPO:

```bash
# DO NOT do this
CUDA_VISIBLE_DEVICES=2 python run.py --mode train ...
CUDA_VISIBLE_DEVICES=4 python run.py --mode train ...
```

The core issue is **policy desynchronisation**, not just a per-group statistical
problem. After the first `optimizer.step()`, each process holds a different set
of weights (`θ_A` on GPU 2, `θ_B` on GPU 4). From that point the two processes
are no longer training the same policy — they have silently diverged into two
independent models. Any subsequent rollouts, rewards, and gradients belong to
different distributions and cannot be meaningfully combined or compared. We are
no longer doing online RL on one policy; we are doing two separate, uncoordinated
experiments.

Contrast with supervised fine-tuning, where data are fixed and independent of
the model. In GRPO, rollouts are generated by the current policy at every step,
so the policy must be **identical across all workers at the start of each step**.

---

## Correct approach: synchronous DDP via Accelerate

The only supported strategy for the first distributed version of this custom
trainer is **synchronous gradient synchronisation** with all ranks staying on
the same `θ`:

```
Step N:
  rank 0 → rollouts for task-shard-0 from θ_t → local loss → local gradient
  rank 1 → rollouts for task-shard-1 from θ_t → local loss → local gradient
           ↓ all-reduce (DDP / NCCL, inside accelerator.accumulate)
  θ_{t+1} = θ_t − lr × mean(grad_rank0, grad_rank1)   ← identical on all ranks

Step N+1:
  both ranks generate rollouts from the same θ_{t+1}   ✓ on-policy
```

Launch command:

```bash
accelerate launch --multi_gpu --num_processes 2 \
  experiments/nestful_mtgrpo_minimal/run.py \
  --mode train \
  --config experiments/nestful_mtgrpo_minimal/config.yaml
```

`--num_processes` = number of distributed training processes, typically one per
GPU. Accelerate maps each process to a separate CUDA device automatically.

## Design principles (first version)

1. **Single-GPU default is preserved.** If `accelerate launch` is not used the
   code falls back to the current single-process path unchanged.
2. **Multi-GPU is explicitly opt-in** — launched via `accelerate launch`, not a
   config flag.
3. **No independent instances.** There is no supported path that runs two
   separate `run.py` processes without DDP gradient sync.
4. **No checkpoint merging.** Merging or averaging LoRA adapters from independent
   runs is not supported and not planned.
5. **One complete GRPO group per rank.** All `num_generations` rollouts for a
   single task stay on one rank. The group advantage is computed locally. Ranks
   never share partial groups.
6. **Balanced task shards.** All ranks must perform the same number of
   backward/optimizer sync steps. If `len(tasks)` is not evenly divisible by
   `world_size`, the task list must be trimmed or padded before sharding so that
   every rank produces the same number of accumulation windows. Uneven shards
   cause a deadlock or gradient corruption in DDP.

## QLoRA + DDP — important constraint

When using `device_map="auto"` (current default in `run.py`), HuggingFace
distributes model layers across all visible devices automatically. **This must
not be used in distributed mode.** With `accelerate launch` each process is
already assigned exactly one GPU (`local_process_index`). The model must be
loaded onto that specific device:

```python
# replace device_map="auto" with:
device = accelerator.local_process_index   # e.g. 0 or 1
model = AutoModelForCausalLM.from_pretrained(
    base_model,
    device_map={"": device},              # pin entire model to this rank's GPU
    quantization_config=quant_config,
    ...
)
```

Mixing `device_map="auto"` with DDP wrapping leads to undefined behaviour and
will likely crash or silently produce wrong gradients.

## Task sharding — Variant A (recommended)

Slice the task list by rank index before the training loop, after ensuring
the list length is a multiple of `world_size`:

```python
# grpo_train.py — to be added when implementing
if accelerator is not None and accelerator.num_processes > 1:
    rank  = accelerator.process_index
    world = accelerator.num_processes
    # trim to nearest multiple so all ranks have equal step counts
    trimmed = len(tasks) - (len(tasks) % world)
    tasks = tasks[:trimmed][rank::world]
```

This keeps each task's full `num_generations`-group on a single rank and avoids
any cross-rank rollout dependency.

**Variant B** (Accelerate DataLoader with `accelerator.prepare`) is possible but
adds complexity without benefit here, because we need the guarantee that one task
and all its rollouts remain on one rank. Variant A is explicit and transparent.

## Minimal code changes required

The following changes to `grpo_train.py` and `run.py` are needed.
**None of these are implemented yet.**

### `run.py` — `mode_train`

```python
from accelerate import Accelerator

def mode_train(config: dict) -> int:
    accelerator = Accelerator()          # no-op if not launched with accelerate

    # … load tasks as before …

    model, tokenizer = load_model_and_tokenizer(
        config, None, for_training=True,
        device=accelerator.local_process_index,   # see QLoRA constraint above
    )
    optimizer = torch.optim.AdamW(trainable, lr=lr)
    model, optimizer = accelerator.prepare(model, optimizer)

    from grpo_train import train
    summary = train(config, model, tokenizer, registry, tasks, log_path,
                    vllm_gen=vllm_gen, wandb_run=wandb_run,
                    accelerator=accelerator)

    if accelerator.is_main_process:
        with open(os.path.join(out_dir, "train_summary.json"), "w") as fh:
            json.dump(summary, fh, indent=2)
        _wandb_log_eval(wandb_run, …)
        _wandb_finish(wandb_run)
```

### `grpo_train.py` — training loop

Replace the current `backward / clip / step` block with:

```python
with accelerator.accumulate(model):
    accelerator.backward(pg_loss)
    if accelerator.sync_gradients:
        accelerator.clip_grad_norm_(trainable, max_grad_norm)
    optimizer.step()
    optimizer.zero_grad(set_to_none=True)
```

`accelerator.accumulate(model)` handles both local gradient accumulation and the
DDP sync barrier. The `sync_gradients` guard ensures clipping happens only on the
step where gradients are actually synchronised across ranks.

### Checkpoint saving

Saving must happen only on the main process, and must use `unwrap_model` to
strip the DDP wrapper before calling `save_pretrained`:

```python
if accelerator.is_main_process:
    unwrapped = accelerator.unwrap_model(model)
    unwrapped.save_pretrained(adapter_dir)
    tokenizer.save_pretrained(adapter_dir)

# all ranks wait until the checkpoint is fully written before continuing
accelerator.wait_for_everyone()
```

Without `unwrap_model`, `save_pretrained` may fail or save the DDP-wrapped state
dict instead of the plain LoRA adapter.

## Metrics aggregation

In multi-GPU mode every rank only sees its shard of tasks. The per-epoch summary
(mean reward, fallback flag, etc.) must be merged before writing.

Recommended approach:

1. Each rank writes `outputs/distributed/rank_{rank}_train_summary.json`.
2. **Main process waits for all ranks** via `accelerator.wait_for_everyone()`
   before reading rank files — without the barrier the files may not be fully
   flushed yet.
3. Main process reads all rank files, merges them, and writes the canonical
   `outputs/train_summary.json`.
4. The merged summary should include `distributed_world_size: N` so downstream
   tooling and the curriculum runner can detect distributed runs.

```python
# each rank
with open(f"outputs/distributed/rank_{rank}_train_summary.json", "w") as f:
    json.dump(local_summary, f)

accelerator.wait_for_everyone()   # ← barrier before main process reads files

if accelerator.is_main_process:
    summaries = [json.load(open(f"outputs/distributed/rank_{i}_train_summary.json"))
                 for i in range(world_size)]
    merged = _merge_summaries(summaries)
    merged["distributed_world_size"] = world_size
    with open("outputs/train_summary.json", "w") as f:
        json.dump(merged, f, indent=2)
```

## Curriculum coordination

**Distributed curriculum is not supported yet.** `run_curriculum.sh` must remain
single-GPU until the distributed train summary aggregation above is implemented
and verified end-to-end.

When distributed curriculum is eventually added, the following constraints apply:

- Only the main process reads `train_summary.json` and evaluates the gate
  condition.
- Only the main process saves the best checkpoint and writes stage manifests.
- All other ranks must wait at a barrier after each stage before the next stage
  begins — they must not proceed independently.
- The next stage uses the single checkpoint path written by the main process.

Attempting to run `run_curriculum.sh` with `accelerate launch` before these
constraints are implemented will produce incorrect gate decisions and potentially
diverging checkpoints per rank.

## vLLM

vLLM server mode and Accelerate/DDP distributed training are **two separate and
currently incompatible scaling paths**. vLLM server mode is not simply an
independent HTTP acceleration layer — it maintains its own copy of the model
weights and requires explicit weight synchronisation between the trainer policy
and the inference server after each checkpoint. In a distributed training setup
this synchronisation becomes significantly more complex: it is unclear which rank
owns the server, how to handle LoRA hot-swap across ranks, and how to ensure the
server sees the post-all-reduce weights rather than a single rank's local state.

**Do not combine vLLM server mode with DDP in the first distributed version.**

For the first distributed trainer, keep generation local via
`transformers.generate` (i.e. `hardware.use_vllm: false`). This avoids the vLLM
ownership question entirely. vLLM server/split generation can be added later as a
separate acceleration path once the DDP baseline is validated.

## Recommended validation sequence

Before running a real multi-GPU curriculum:

```
1. Run single-GPU smoke (must still pass unchanged):
   python run.py --mode smoke --config config.yaml

2. Run single-GPU tiny train:
   python run.py --mode train --config config.yaml \
     --override data.max_train_tasks=16 \
     --override training.epochs=1

3. Run 2-GPU tiny train (first DDP test, vLLM disabled):
   accelerate launch --multi_gpu --num_processes 2 \
     run.py --mode train --config config.yaml \
     --override data.max_train_tasks=16 \
     --override training.epochs=1 \
     --override hardware.use_vllm=false

4. Verify train_summary.json contains:
     "distributed_world_size": 2
     "fallback_used": false
     steps count consistent with 16 tasks / 2 ranks / grad_accum

5. Only then run full curriculum with accelerate launch.
```
